RÈGLES FRONTEND

Le frontend doit être :

ultra fluide,
modulaire,
optimisé,
découplé,
responsive,
maintenable.

Obligatoire :

smart hooks + dumb components,
lazy loading,
virtualisation,
optimistic UI,
optimisation des rerenders,
architecture feature-based,
composants réutilisables,
Zustand selectors,
TanStack Query.

Interdictions :

logique métier dans UI,
composants géants,
prop drilling massif,
state spaghetti,
fetch manuel inutile,
useEffect complexes inutiles.
RÈGLES BACKEND

Le backend doit respecter :

controllers ultra fins,
logique métier dans domaine,
validation stricte,
transactions robustes,
gestion concurrence,
orchestration découplée,
side-effects asynchrones.

Aucune logique métier dans :

routes,
Prisma,
DTO,
controllers.
MULTI-TENANT

Toutes les entités doivent contenir :

tenant_id

Aucune requête ne doit bypass :

tenant scope,
RLS,
isolation des données.

Tous les repositories doivent automatiquement :

injecter tenant_id,
empêcher cross-tenant access,
appliquer RLS.

La sécurité multi-tenant est prioritaire.

LOGIQUE MÉTIER CRITIQUE
Disponibilités

Une chambre ne peut jamais :

être doublement réservée,
être désynchronisée,
être disponible incorrectement.

Toute modification doit :

mettre à jour le planning en temps réel,
préserver l’intégrité métier,
gérer les conflits.
Pricing

Les tarifs dépendent :

des dates,
des saisons,
du type de chambre,
du plan tarifaire,
du yield management,
des règles automatiques.

Le moteur tarifaire doit être :

déterministe,
traçable,
performant,
auditables.
Finance

Les données financières sont IMMUTABLES.

Interdiction de modifier :

factures,
paiements,
écritures comptables,
transactions validées.

Utiliser uniquement :

reversal entries,
correction entries,
audit logs.
PERFORMANCE

Le système doit donner une sensation :

instantanée,
fluide,
premium,
sans latence perceptible.

L’utilisateur clique → la réponse doit sembler immédiate.

Obligatoire :

cache intelligent,
optimisation DB,
optimisation réseau,
optimistic UI,
lazy loading,
virtualisation,
pagination,
background jobs.

Interdictions :

moulinage inutile,
re-renders massifs,
requêtes non bornées,
polling abusif,
composants géants,
traitements synchrones lourds.

les APIs sont ciblées,

Obligatoire :

validation stricte,
auth robuste,
RBAC,
protection injections,
rate limiting,
audit logging,
sanitation,
contrôle permissions.

Interdictions :

secrets hardcodés,
endpoints non sécurisés,
confiance frontend,
accès non contrôlés.
